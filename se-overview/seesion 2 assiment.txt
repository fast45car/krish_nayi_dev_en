The Three Main Layers of Software Architecture


While there are many ways to slice up an architecture, most layered software systems converge on three core layers, often called the 3-tier or 3-layer architecture:
1. Presentation Layer (UI Layer)

This is what the user sees and interacts with directly.

Handles user input and displays output
Examples: web pages, mobile app screens, desktop GUIs, command-line interfaces
Should contain minimal logic — mostly formatting, validation of input format, and routing user actions to the next layer
Technologies: HTML/CSS/JS, React, Swift/Kotlin, Tkinter, etc.

2. Business Logic Layer (Application/Domain Layer)

This is the "brain" of the application — where the actual rules and processes live.

Contains the core logic: calculations, decision-making, workflows, validations
Enforces business rules (e.g., "an order can't ship without payment confirmation")
Coordinates data flow between the presentation layer and data layer
Should be independent of how data is displayed or stored, so it can be reused/tested in isolation

3. Data Layer (Persistence/Data Access Layer)

This layer manages how data is stored, retrieved, and persisted.

Talks to databases, file systems, or external APIs
Handles queries, transactions, and data mapping (e.g., ORM)
Abstracts storage details away from the business logic — so swapping a database (e.g., MySQL → PostgreSQL) shouldn't require rewriting business rules
Examples: SQL/NoSQL databases, ORMs (Hibernate, SQLAlchemy), file storage, caching layers

## Client and Server

**Client** is the program or device that *requests* something — it's what the user directly interacts with. It asks for data or services but usually doesn't store or process things on its own.

**Server** is the program or machine that *listens* for those requests and *responds* with the requested data or service. It does the heavy lifting — storing data, running logic, and sending back results.

In short: **the client asks, the server answers.**

---

### Real-Life Example: Ordering at a Restaurant

Think of it like a restaurant:

- **You (the client)** sit at a table and tell the waiter what you want — "I'll have the pasta."
- **The kitchen (the server)** receives that order, prepares the food using its ingredients and recipes, and sends the finished dish back out to you.

You don't cook the food yourself, and the kitchen doesn't decide what you eat — it just *responds* to your request. If ten other tables order food at the same time, the kitchen handles all those requests too, just like a server handles requests from many clients (browsers, apps, etc.) at once.

### Tech Equivalent
When you open a website like `www.amazon.com`:
- Your **browser** (client) sends a request: "Give me the homepage."
- Amazon's **server** processes that request, pulls the right data (products, prices, your cart), and sends back the webpage you see.





Differences: Desktop Apps vs Web Apps vs Mobile Apps


| Feature         | Desktop App                         | Web App            | Mobile App                      |
| --------------- | ----------------------------------- | ------------------ | ------------------------------- |
| Runs On         | PC/Laptop                           | Browser            | Smartphone/Tablet               |
| Installation    | Required                            | Not required       | Required from app store         |
| Internet Needed | Often not                           | Usually yes        | Depends on app                  |
| Performance     | Fastest                             | Moderate           | Fast                            |
| Access Hardware | Full access                         | Limited            | Good access (camera, GPS, etc.) |
| Updates         | User installs                       | Automatic          | Through app store               |
| Examples        | Visual Studio Code, Adobe Photoshop | Gmail, Google Docs | WhatsApp, Instagram             |
